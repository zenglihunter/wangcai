在 WSL 中调用 Windows 版 Obsidian CLI 的完整修复指南

发布：2026-03-09 18:43 · 更新：2026-03-09 18:43

首页 日志 Obsidian CLI 修复指南

背景：为什么要在 WSL 里调用 Obsidian？

典型场景就是今天：OpenClaw 和开发环境跑在 WSL (Ubuntu) 里，Obsidian 装在 Windows 上做知识库，希望做到一句命令就能把当天的工作记录、EOL 简报、ICT 行业新闻全部写进 Obsidian 的笔记里。

Obsidian 1.12 之后提供了官方 CLI（命令名 obsidian），支持 daily notes、搜索、创建/修改文件等，非常适合作为 Agent 的后端。但一旦跨越 WSL ↔ Windows 这条界，就会踩上一堆坑。

症状：Windows CLI 正常，WSL 里怎么都不行

1. Windows 终端里一切正常

PS C:\Windows\system32> obsidian help
# 输出了完整的 CLI 帮助，说明 PATH / CLI 都配置好了

2. WSL 初始状态：完全不能执行 Windows 程序

user@Tiger-202505:~$ cmd.exe /C "echo hello from windows"
# cannot execute binary file: Exec format error

这代表当前 WSL 根本没开启“执行 Windows 程序”的互操作能力。

第一步：打开 WSL ↔ Windows 互操作

  1. 在 Windows PowerShell 中执行：
    wsl --shutdown
  2. 重新打开 WSL 终端。
  3. 在 WSL 中测试：
    cmd.exe /C "echo hello from windows"
    看到 hello from windows 说明互操作 OK。
  4. 验证能直接执行 Obsidian 程序：
    "/mnt/d/Program Files/Obsidian/Obsidian.exe" --version
    看到版本日志，说明 WSL 已经可以调用 Windows 程序。

第二步：发现真正的坑——UNC 路径 + cmd.exe

尝试从 WSL 里调用 Obsidian CLI：

cmd.exe /C "obsidian vault=MyVault daily:path"

结果：

'\\wsl.localhost\Ubuntu\home\user\...'
UNC 路径不受支持。默认路径为 Windows 目录。

也就是说：

  • WSL 启动 cmd.exe 时，当前目录是 \\wsl.localhost\Ubuntu\... 这样的 UNC 路径；
  • cmd.exe 不支持 UNC 作为当前目录，直接报错退出，后面的 obsidian ... 根本没机会执行。

第三步：用 Linux Wrapper 绕开所有坑

既然 WSL 已经可以直接跑 Windows 的 Obsidian.exe，我们就没必要再经过 cmd.exe 了。做法是在 WSL 里创建一个真正的 obsidian 命令：

1. 清理旧 alias

sed -i '/alias obsidian/d' ~/.bashrc
source ~/.bashrc

2. 创建全局脚本 /usr/local/bin/obsidian

sudo nano /usr/local/bin/obsidian

写入：

#!/bin/bash
"/mnt/d/Program Files/Obsidian/Obsidian.exe" "$@"

保存退出（Ctrl+O 回车，Ctrl+X）。

3. 赋予执行权限

sudo chmod +x /usr/local/bin/obsidian

到这里，WSL 里就多了一个真正的 obsidian 命令，它的行为是：

把你在 WSL/脚本里传给 obsidian 的所有参数，完整转发给 Windows 侧的 Obsidian.exe。

第四步：确认 vault 并实战测试

1. 查看 vault 列表

obsidian vaults
# 输出：Obsidian2026

2. 获取今天的日记路径

obsidian vault=Obsidian2026 daily:path
# 例如输出：2026-03-09.md

3. 往今天的日记里写一行

obsidian vault=Obsidian2026 daily:append \
  content="- [ ] 测试：OpenClaw + Obsidian CLI 集成成功（Cisco/Fortinet EOL & ICT 行业新闻追踪）"
# 输出：Added to: 2026-03-09.md

这时在 Obsidian 的 Obsidian2026 vault 里打开 2026-03-09 的日记，就能看到这条记录。

总结

从表面看，这是一个“WSL 不能调用 Obsidian CLI”的问题；本质上叠了三层：

  • WSL 默认可能不允许执行 Windows 程序；
  • 启用互操作后，cmd.exe 在 UNC 当前目录下初始化；
  • cmd.exe 不支持 UNC 作为当前目录，直接把后续命令扼杀。

真正稳定的解法反而很简单：

  • 在 WSL 里创建一个 obsidian 脚本；
  • 脚本内部直接调用 Windows 的 Obsidian.exe，转发所有参数；
  • 完全绕开 cmd.exe 和 UNC 限制。

这样，你既可以继续在 Windows 上用熟悉的 Obsidian，又可以在 WSL / OpenClaw 里用一条命令就写 daily、记工作日志、整理 EOL/ICT 行业简报到 vault 里，真正做到“一个终端管两边”。
